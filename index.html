flask-上下文管理

发表于 2018-12-06 | 分类于 flask

上下文

请求上下文和应用上下文

关于上下文的理解,借用知乎轮子哥的一个回答

每一段程序都有很多外部变量。只有像Add这种简单的函数才是没有外部变量的。一旦你的一段程序有了外部变量，这段程序就不完整，不能独立运行。你为了使他们运行，就要给所有的外部变量一个一个写一些值进去。这些值的集合就叫上下文。

应用上下文可以保持应用级数据在整个请求过程或者在shell中可用,而不是通过把应用当做参数传递给每个函数.应用级数据可以通过current_app和g获取.

在flask中,视图函数需要知道它执行情况的请求信息（请求的 url，参数，方法等）以及应用信息（应用中初始化的数据库等），才能够正确运行。

最直观地做法是把这些信息封装成一个对象，作为参数传递给视图函数。但是这样的话，所有的视图函数都需要添加对应的参数，即使该函数内部并没有使用到它。

flask 的做法是把这些信息作为类似全局变量的东西，视图函数需要的时候，可以使用 from flask import request 获取。但是这些对象和全局变量不同的是——它们必须是动态的，因为在多线程或者多协程的情况下，每个线程或者协程获取的都是自己独特的对象，不会互相干扰。

flask 核心依赖

Flask 虽然是一个微型的Web框架，但是包含了web框架所必须的核心功能。Flask框架主要依赖两个库。

  • Werkzeug ： 包含了路由、WSGI、调试的工具集。负责核心的逻辑模块。
  • Jinja2 ：flask默认的模板引擎(可以替换为其他的，比如Mako).负责对模板的渲染。

Werkzeug

werkzeug为flask提供的功能：

  1. 路由处理：怎么根据请求的url找到对应的view（视图）处理函数
  2. 对request和response的封装：可以更好的读取request的数据，也容易生成响应
  3. 自带了一个WSGI server， 可以用来测试环境运行自己的应用

mysql-高级操作1-函数的使用

发表于 2018-10-04 | 分类于 数据库

使用函数处理数据

与其他大多数计算机语言一样, SQL支持利用函数来处理数据。函数一般是在数据上执行的,它给数据的转换和处理提供了方便。

局限性

  1. 函数没有SQL的可移植性强能运行在多个系统上的代码称为可移植的(portable)。相对来说,多数SQL语句是可移植的,在SQL实现之间有差异时,这些差异通常不那么难处理。而函数的可移植性却不强。几乎每种主要的DBMS的实现都支持其他实现不支持的函数,而且有时差异还很大。

  2. 为了代码的可移植,许多SQL程序员不赞成使用特殊实现的功能。虽然这样做很有好处,但不总是利于应用程序的性能。如果不使用这些函数,编写某些应用程序代码会很艰难。必须利用其他方法来实现DBMS非常有效地完成的工作。

  3. 如果你决定使用函数,应该保证做好代码注释,以便以后你(或其他人)能确切地知道所编写SQL代码的含义

TCP/IP 协议

协议是指计算机与网络设备要相互通信,双方必须基于相同的方法.比如,如何探测到通信目标,哪一方先发起通信,使用那种语言进行通讯,怎样借宿通信等等规则.不同的硬件.操作系统之间的通信都需要一种规则,这种规则就称为协议(protocol)

TCP/IP协议是互联网相关的各种协议的总称.

分层管理

TCP/IP协议族中最重要的一点就是分层管理.TCP/IP协议按层次分为:网络链路层, 网络层, 传输层, 应用层.

各层次的作用:

  1. 链路层(也称链路层, 网络接口层)

用来处理链接网络的硬件部分.包括控制操作系统,硬件的设备驱动, 网卡, 光纤等物理课件的部分.硬件上的范畴均在链路层的作用范围.

werkzeug计算密码散列值

在数据库中保存用户密码时,处于安全考虑,我们储存的用户密码加密后的密码散列值.

werkzeug中的security模块中提供的两个方法可以很方便的实现这个功能.

  1. generate_password_hash(password, method=pdkdf2:sha1, salt_length=8), 该方法输入为原始密码,输出是字符串形式的密码散列值.可以用在用户注册的时候,数据库中保存用户密码散列值.method, salt_length的默认值就能满足大部分需求.
  2. check_password_hash(hash, password),这个函数的参数是数据库中存储的密码散列值和用户提供的密码,两者进行核对,如果密码正确则返回True,否则,返回False.
