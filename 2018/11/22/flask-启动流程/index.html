flask-启动流程

发表于 2018-11-22 | 分类于 flask

flask 核心依赖

Flask 虽然是一个微型的Web框架，但是包含了web框架所必须的核心功能。Flask框架主要依赖两个库。

  • Werkzeug ： 包含了路由、WSGI、调试的工具集。负责核心的逻辑模块。
  • Jinja2 ：flask默认的模板引擎(可以替换为其他的，比如Mako).负责对模板的渲染。

Werkzeug

werkzeug为flask提供的功能：

  1. 路由处理：怎么根据请求的url找到对应的view（视图）处理函数
  2. 对request和response的封装：可以更好的读取request的数据，也容易生成响应
  3. 自带了一个WSGI server， 可以用来测试环境运行自己的应用

Jinja2

Jinja2 是Flask的默认引擎，功能非常丰富，支持unicode解析、自动HTML escape防止XSS攻击，继承、变量、过滤器、流程逻辑支持、python代码逻辑继承等等。

Flask应用启动流程

分析思路：首先看一个最小的应用开始，然后来分析一个应用程序都用到了Flask的那些功能，最后具体对应源码找到功能是如何实现的。

下面先看号称世界上最小的应用：

from flask import Flask

app = Flask(__name__)


@app.route('/')
def hello_world():
    return 'Hello World！'


if __name__ == '__main__':
    app.run()

整体来看，这段代码，只是建立了一个Flask()类的实例，并且传入了一个参数name.这个便是WSGI web应用部分，web应用实际上就是Flask类的实例。然后使用app.route装饰器实现了路由和view视图函数的匹配关系。最后通过app.run()实现web服务的启动。

下面我们分别看看各功能是怎么实现的：

  • app = Flask(name), 建立Flask实例的时候，为什么要传入name的参数呢，这个参数有什么作用，带着这个问题，找下Flask这个类的实现过程。

在源码flask文件夹的app.py文件中，定义了这个Flask核心的类Flask类。

class Flask(_PackageBoundObject):
    def __init__(self, import_name,
                 static_path=None,
                 static_url_path=None,
                 static_folder='static', 
                 template_folder='templates',
                 instance_path=None,
                 instance_relative_config=False,
                 root_path=None):
        _PackageBoundObject.__init__(self, import_name,
                                     template_folder=template_folder,
                                     root_path=root_path)

class _PackageBoundObject(object):
    def __init__(self, import_name, template_folder=None, root_path=None):
        #: The name of the package or module.  Do not change this once
        #: it was set by the constructor.
        self.import_name = import_name

        #: location of the templates.  ``None`` if templates should not be
        #: exposed.
        self.template_folder = template_folder

        if root_path is None:
            root_path = get_root_path(self.import_name)

        #: Where is the app root located?
        self.root_path = root_path

从源码中可见， Flask类继承_PachageBoundObject类，只有一个必要的参数，就是import_name,他接受包或者模块的名字，但是一般传递的是name.然后helpers.get_route_path函数，通过传入的这个参数确定程序的根目录，以便获取静态文件和模板文件的目录。

  • app.route装饰器会将url和执行的视图函数的关系保存到app.url_map属性上。以便根据请求的url找到对应的视图函数。这里是将’\’和视图函数hello_word相关联。

  • app.run().这个便是应用程序启动的命令。下面从源码的角度看下这个命令背后的过程。

启动过程

我们从源码的flask.app.Flask中找到run()方法的定义。下面展示的源码是去掉相关注释后的代码。

def run(self, host=None, port=None, debug=None, **options):
    from werkzeug.serving import run_simple
    if host is None:
        host = '127.0.0.1'
    if port is None:
        server_name = self.config['SERVER_NAME']
        if server_name and ':' in server_name:
            port = int(server_name.rsplit(':', 1)[1])
        else:
            port = 5000
    if debug is not None:
        self.debug = bool(debug)
    options.setdefault('use_reloader', self.debug)
    options.setdefault('use_debugger', self.debug)
    try:
        run_simple(host, port, self, **options)
    finally:
        self._got_first_request = False

从上述代码中可以看到，调用app.run(),这个函数比较简单，只是对传入的参数进行处理，如果没有传入host、port、dubug参数，那么Flask会默认使用host=127.0.0.1,port=5000, use_reloader=False, use_debugger=Flase.最后真正的启动服务是通过调用werkzeug.serving.run_simple()方法启动服务，接受的参数就是run()方法处理过的host和port，值得注意的是，run_simple()的第三个参数，他接受的是Flask实例本身。

我们进一步看下run_simple()方法：

def run_simple(hostname, port, application, use_reloader=False,
           use_debugger=False, use_evalex=True,
           extra_files=None, reloader_interval=1,
           reloader_type='auto', threaded=False,
           processes=1, request_handler=None, static_files=None,
           passthrough_errors=False, ssl_context=None):

    if not isinstance(port, int):
        raise TypeError('port must be an integer')
    if use_debugger:
        from werkzeug.debug import DebuggedApplication
        application = DebuggedApplication(application, use_evalex)
    if static_files:
        from werkzeug.wsgi import SharedDataMiddleware
        application = SharedDataMiddleware(application, static_files)

    def log_startup(sock):
        display_hostname = hostname not in ('', '*') and hostname or 'localhost'
        if ':' in display_hostname:
            display_hostname = '[%s]' % display_hostname
        quit_msg = '(Press CTRL+C to quit)'
        port = sock.getsockname()[1]
        _log('info', ' * Running on %s://%s:%d/ %s',
             ssl_context is None and 'http' or 'https',
             display_hostname, port, quit_msg)

    def inner():
        try:
            fd = int(os.environ['WERKZEUG_SERVER_FD'])
        except (LookupError, ValueError):
            fd = None
        srv = make_server(hostname, port, application, threaded,
                          processes, request_handler,
                          passthrough_errors, ssl_context,
                          fd=fd)
        if fd is None:
            log_startup(srv.socket)
        srv.serve_forever()

    if use_reloader:
        if os.environ.get('WERKZEUG_RUN_MAIN') != 'true':
            if port == 0 and not can_open_by_fd:
                raise ValueError('Cannot bind to a random port with enabled '
                                 'reloader if the Python interpreter does '
                                 'not support socket opening by fd.')

            address_family = select_ip_version(hostname, port)
            s = socket.socket(address_family, socket.SOCK_STREAM)
            s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
            s.bind(get_sockaddr(hostname, port, address_family))
            if hasattr(s, 'set_inheritable'):
                s.set_inheritable(True)

            if can_open_by_fd:
                os.environ['WERKZEUG_SERVER_FD'] = str(s.fileno())
                s.listen(LISTEN_QUEUE)
                log_startup(s)
            else:
                s.close()

        from werkzeug._reloader import run_with_reloader
        run_with_reloader(inner, extra_files, reloader_interval,
                          reloader_type)
    else:
        inner()

上述代码中，比较核心的服务是inner()方法，该方法中通过make_server()方法来创建相应的WSGIServer。默认情况下使用的是单线程的werkzeug.serving.BaseWSGIServer处理请求，实际上还是使用的标准库的BaseHTTPServer.HTTPServer。

本节总结：通过上述方法，再启动app.run()方法候,通过调用werkzeug.server.run_simple方法，默认情况下建立一个BsseWSGIServer监听端口，接受客户端发送过来的请求,这样一个web应用服务便启动了，会默认监听127.0.0.1:5000端口。

那么当一个HTTP请求过来，是怎么进行处理的？下面分析下这个过程。

处理HTTP请求

通过上面使我们知道app.run()启动一个web服务器,用来监听5000端口.下面便来分析下,当客户端请求过来候,Werkzeug是怎么处理请求,又是通过什么方法实现了WSGI的功能(将请求转化成flask应用能够处理的请求), 又是怎么将相应返回给客户端的.

上面提到app.run()启动后,会启动一个BaseWSGIServer对象的web服务器.这个服务器对象在werkzeug.serving.BaseWSGISERver中定义,下面是源码的定义:

BaseWSGIServer类用werkzeug.serving.WSGIRequestHandler类替换了HTTPServer默认的处理请求方法.而这个类就是werkzeug实现WSGI功能的关键.负责WSGI的调度.

BaseWSGIServer类是用来处理客户端的请求, 包装相关参数和生成start_response参数.也就是把请求处理成符合WSGI格式的,有关WSGI相关的知识可以看这篇博客,写的很棒.然后调用flask应用,也就是我们生成的Flask实例app, app.(也就是run()的app.)

下面是该类的源码的部分片段,我们关注和Flask的实例app有关的片段:

def execute(app):
    application_iter = app(environ, start_response)  # 调用app, app是Flask类的实例
    try:
        for data in application_iter:
            write(data)
        if not headers_sent:
            write(b'')
    finally:
        if hasattr(application_iter, 'close'):
            application_iter.close()
        application_iter = None

这个方法中调用了app(), 而app是Flask类的实例,一个类如果能被直接调用,那么这个类必须实现call方法.实现call方法的类,是一个可调用对象,直接调用类,会自动运行call方法.我们返回到Flask()类中,看看call方法返回的对象.

def __call__(self, environ, start_response):
"""Shortcut for :attr:`wsgi_app`."""
return self.wsgi_app(environ, start_response)

返回的是.app.wsgi_app()方法的结果.顺着找到wsgi_app()方法:

def wsgi_app(self, environ, start_response):
    ctx = self.request_context(environ)
    ctx.push()
    error = None
    try:
        try:
            response = self.full_dispatch_request()
        except Exception as e:
            error = e
            response = self.handle_exception(e)
        return response(environ, start_response)
    finally:
        if self.should_ignore_error(error):
            error = None
        ctx.auto_pop(error)

wsgi_app(environ, start_response)方法是真正的WSGI应用,environ参数是wsgi环境参数,HTTP请求的参数可以从里获得,start_response参数是一个可调用对象,接受一个状态码,一个headers的列表和一个可选的异常文本最为响应.该方法的处理过程是:调用request_context()方法,创建一个 RequestContext(environ)请求上下文对象,并把该对象压入_request_ctx_stack栈中,然后调用full_dispatch_request()方法对请求进行分发.进一步处理请求,下一步分析该方法.在外层try…finally…可以看到,在处理完请求(不管是正确处理还是出现异常)最后ctx.auto_pop(error)最后该请求都会从_request_ctx_stack中出栈.
把这个方法不直接写到call方法中,这么做是为了使用中间件的时候不会失去对Flask类的引用,当中间件调用应用的时候可以这么写:

app.wsgi_app = MyMiddleware(app.wsgi_app)

这样仍然可以获原始的应用对象,并且可以继续调用它的方法.

Flask.full_dispatch_request()方法源码:

def full_dispatch_request(self):
    self.try_trigger_before_first_request_functions()
    try:
        request_started.send(self)
        rv = self.preprocess_request()
        if rv is None:
            rv = self.dispatch_request()
    except Exception as e:
        rv = self.handle_user_exception(e)
    return self.finalize_request(rv)

def try_trigger_before_first_request_functions(self):

    if self._got_first_request:
        return
    with self._before_request_lock:
        if self._got_first_request:
            return
        for func in self.before_first_request_funcs:
            func()
        self._got_first_request = True

def dispatch_request(self):
    req = _request_ctx_stack.top.request
    if req.routing_exception is not None:
        self.raise_routing_exception(req)
    rule = req.url_rule
    # if we provide automatic options for this URL and the
    # request came with the OPTIONS method, reply automatically
    if getattr(rule, 'provide_automatic_options', False) \
       and req.method == 'OPTIONS':
        return self.make_default_options_response()
    # otherwise dispatch to the handler for that endpoint
    return self.view_functions[rule.endpoint](**req.view_args)

执行逻辑:

  1. 首先执行Flask.try_trigger_before_first_request_functions(), 该函数在每次请求之前调用Flask.before_first_request_funcs列表中的函数,并且每个Flask实例在每次请求只能调用一次.

  2. Flask.preprocess_request()和Flask.finalize_request()方法里面包含了请求处理之前和之后的很多hooks(钩子), 包括

    • 第一次请求处理之前的 hook 函数，通过 before_first_request 定义
    • 每个请求处理之前的 hook 函数，通过 before_request 定义
    • 每个请求正常处理之后的 hook 函数，通过 after_request 定义
    • 不管请求是否异常都要执行的 teardown_request hook 函数

  3. 其中dispatch_request()是调用与请求相对应的视图函数,并返回结果的真正的执行函数.这个函数的主要逻辑是通过请求的url,找到对应的视图函数,然后返回响应.

  4. 最后通过finalize_request()对返回的response进行包装成Response对象返回客户端.
